# 系统一致性修改文档 v1.0

> 生成时间: 2026-06-01 | 基于 PRD v3.0 + 实施计划 v1.17 + 系统设计方案 v2.9 + 当前代码库扫描

## 一、整体结论

当前代码库在**核心业务流程**（用户认证、商品浏览、购物车、订单、基础支付）上已跑通，但在**基础设施层**和**高级业务功能**上与 PRD/设计方案存在显著差距。主要体现为：

- **设计文档要求 116 张表 / 42 套枚举 / 30 Events + 80 Listeners / Actions+Queries+Jobs+Commands 齐备**
- **实际代码 53 张表 / 少量枚举 / 1 Event + 2 Listener / Actions+Jobs+Commands 全为空**

## 二、详细差距清单

### 2.1 后端 — 基础设施层（P0）

| # | 缺失项 | 设计方案要求 | 当前状态 | 影响 |
|---|--------|-------------|---------|------|
| 1 | `BaseAction` | Phase 1 要求所有业务操作封装为 Action | 不存在 | Controller 臃肿，复用性差 |
| 2 | `BaseJob` / 队列 | Phase 1 要求异步处理通知/日志/导出 | 目录不存在 | 同步执行阻塞主线程 |
| 3 | `BaseCommand` | Phase 1 要求 Artisan 命令基类 | 目录不存在 | 无法执行定时任务/数据修复 |
| 4 | `RateLimiter` | API 规范要求敏感接口限流 | 不存在 | 登录/支付/上传无防刷 |
| 5 | `LockManager` | 系统设计要求分布式锁 | 不存在 | 库存/优惠券/支付回调竞态风险 |
| 6 | `Outbox` 模式 | 系统设计要求事件可靠投递 | 不存在 | 跨 Domain 数据一致性风险 |
| 7 | `StateMachine` | 方案要求订单/售后/退款状态机 | 目录存在但为空 | 状态流转全靠手动判断 |
| 8 | `Event/Listener` | 30 Events + 80 Listeners | 1+2 | 业务耦合，通知/日志内嵌 Controller |

### 2.2 后端 — 数据层（P0-P1）

当前 53 migrations，PRD 要求 116 张核心表，**缺口约 63 张**。以下为关键缺失表（按业务优先级排序）：

| 表名 | 所属业务 | 影响 |

| `sub_accounts` | 企业认证 | 子账号体系缺失 |
| `platform_shops` | 商城 | 店铺入驻/管理无表支撑 |
| `stores` | 商城 | 门店体系缺失 |
| `order_files` | 订单 | 印刷文件上传/审核无记录表 |

| `payment_logs` | 支付 | ✅ 已存在（2026-06-06 创建） |

### 2.3 后端 — API 层（P1-P2）

PRD 要求 141 个 API，当前已实现的约 60+，**缺口约 80 个**。关键缺失：

- 企业子账号 CRUD 全套
- 店铺入驻/管理/审核
- 生产排程查询/更新
- 印刷计价详情/历史
- 文件上传/审核/预览
- 消息通知已读/删除/批量操作
- 数据看板统计（多维度）

### 2.4 前端 — 运行时缺陷（P0）

| # | 问题 | 位置 | 影响 |
|---|------|------|------|
| 1 | `fetchUserInfo()` 方法未定义但被调用 | `MainLayout.vue:49` | 页面刷新后白屏/控制台报错 |
| 2 | `composables/` 完全为空 | `frontend/src/composables/` | 大量页面重复分页/表单逻辑 |
| 3 | 组件库仅 4 个 | `frontend/src/components/` | 开发效率低，UI 一致性差 |
| 4 | Admin 视图普遍 <150 行 | `frontend/src/views/admin/` | 后台功能深度严重不足 |

### 2.5 测试基础设施（P1）

| # | 问题 | 影响 |
|---|------|------|
| 1 | Vitest stubs 简陋 | 大量 Element Plus 组件被 mock 为 `<div/>`，无法验证交互 |
| 2 | 无 E2E 测试 | 核心流程无端到端保障 |

## 四、TDD 执行纪律

1. **每次 Batch 开始前**: `git push origin main` 确保代码已同步
2. **红**: 先写测试，确保全部失败
3. **绿**: 实现代码，确保全部通过
4. **重构**: 优化代码结构，测试仍通过
5. **回归**: 运行全量测试（后端 + 前端），确认零回归
6. **提交**: `git add -A && git commit -m "Batch X: ..." && git push`
