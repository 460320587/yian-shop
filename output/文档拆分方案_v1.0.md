# 怡安印刷商城系统设计方案 — 文档拆分方案 v1.0

> **拆分目标**：将当前 ~100,600 行的单体文档，按"架构层次 + 领域边界 + 读者角色"三维度拆分，使单文件控制在 2,000~4,000 行，支撑 20+ 人团队的并行开发与独立演进。

---

## 一、当前问题诊断

| 问题 | 影响 | 严重程度 |
|------|------|:--------:|
| 单文件 100,600 行 | IDE 卡顿、搜索慢、Git diff 不可读 | 🔴 P0 |
| 22 个业务模块揉在一起 | 多人编辑同一文件，冲突率高 | 🔴 P0 |
| 架构总纲与代码实现混杂 | 新成员无法快速定位需要阅读的内容 | 🟡 P1 |
| API 契约与实现细节耦合 | 前端/测试无法独立获取接口文档 | 🟡 P1 |
| 审查映射表占 10,000+ 行 | 污染主干设计文档的可读性 | 🟡 P1 |
| 数据库设计与业务逻辑交错 | DBA 与后端开发关注点混杂 | 🟢 P2 |

## 二、拆分原则（顶级架构师视角）

### 原则 1：按"稳定性"分层
- **稳定层**：架构决策、技术选型、分层约定 —— 变更周期以月/季度计
- **半稳定层**：数据库 schema、领域模型、枚举定义 —— 变更周期以周计
- **活跃层**：Controller/Action/Service 代码、API 路由 —— 变更周期以天计
- **附属层**：审查报告、FR 映射表、覆盖度统计 —— 变更一次性的或周期性批量更新

### 原则 2：按"读者角色"切分
| 角色 | 关注内容 | 目标文件 |
|------|----------|----------|
| 技术总监/架构师 | 架构决策、非功能性指标、技术选型 | `01-架构总纲.md` |
| 后端开发 | 领域模型、业务规则、代码实现 | `06-XX-领域模块.md` |
| 前端开发 | API 契约、Resource 字段、错误码 | `90-API契约手册.md` |
| DBA/运维 | 数据库设计、索引、归档、部署 | `05-数据库设计.md` + `99-部署运维.md` |
| 测试/QA | FR 映射、用例覆盖度、验收标准 | `95-审查报告汇总.md` |

### 原则 3：单文件行数控制
- **上限**：4,000 行（GitHub/Web IDE 可流畅渲染的阈值）
- **理想**：2,000~3,000 行
- **例外**：数据库设计可放宽至 5,000 行（表结构天然密集）

### 原则 4：领域边界优先于技术层次
- 同一领域的 Migration + Model + Controller + Service 应放在同一文件
- 避免"按技术层次拆分"导致的"改一个需求要改 4 个文件"问题
- 例外：通用基础设施（缓存、队列、日志）独立成文件

## 五、拆分实施步骤（渐进式，零停机）

### Phase 1：分离附属层（1天，零风险）
1. 将 FR 映射表、审查报告从主文档剪切到 `95-XX` 系列文件
2. 主文档从 ~100,600 行降至 ~90,000 行
3. **立即收益**：开发人员不再被审查元数据干扰

### Phase 2：分离稳定层（1天，低风险）
1. 将第1~4章（架构总纲、技术栈、分层规范、DDD约定）提取为独立文件
2. 主文档保留为"业务模块合集 + 数据库设计"
3. **立即收益**：新人入职只需读 4 个文件即可理解架构

### Phase 3：按模块拆分活跃层（3~5天，需要协调）
1. 将 06 章的 22 个子模块逐一提取为独立文件
2. 每个模块文件内包含：Migration（引用或内联）+ Model + Controller + Action + Service
3. **关键决策**：数据库 Migration 放在 `05-数据库设计.md` 还是各模块文件？
   - **建议**：通用表（customers/products/orders）放在 `05`，领域专属表放在模块文件
4. **立即收益**：10 个开发可并行修改 10 个模块文件，零冲突

### Phase 4：提取 API 契约层（1天，需要前端配合）
1. 将所有 Resource/Request/路由表汇总到 `90-API契约手册.md`
2. 与前端团队约定：此后接口变更只需更新此文件
3. **立即收益**：前后端解耦，接口文档即代码

### Phase 5：建立交叉引用（持续）
1. 在每个文件头部添加 `"本文件引用"` 和 `"被引用"` 清单
2. 使用 Markdown 锚点实现跨文件跳转（`[订单状态机](../06-04-订单与状态机.md#状态机)`）
3. **长期收益**：形成可导航的知识图谱

## 六、关键决策与折中

### 决策 1：数据库 Migration 放在哪里？
| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|:------:|
| A：全部集中在 05 | DBA 一次看完所有表 | 改模块需跨文件 | ⭐⭐⭐ |
| B：分散在各模块 | 改一个模块只改一个文件 | 全库视图分散 | ⭐⭐⭐⭐ |
| C：双写（05 放摘要，模块放详情） | 兼顾两者 | 维护成本高 | ⭐⭐ |

**推荐 B**：按模块拆分，但在 `05` 中保留一张"全库表清单索引"

### 决策 2：Controller 代码放在模块文件还是 API 契约文件？
**推荐**：Controller 实现放在模块文件，接口契约（Resource/Request）放在 `90`
- 理由：后端开发关注实现，前端关注契约，两者变更节奏不同

### 决策 3：审查报告是否纳入版本控制？
**推荐**：纳入，但放在 `95-审查报告/` 子目录，与主干设计文档分离
- 理由：审查历史是知识资产，但不应污染当前设计视图

### 决策 4：是否使用文档生成工具（如 MkDocs/VuePress）？
**推荐**：Phase 3 完成后引入
- 短期：纯 Markdown 文件，Git 管理即可
- 中期：引入 MkDocs 生成静态站点，支持全文搜索
- 长期：接入 CI，自动生成 API 文档（Swagger/OpenAPI）
