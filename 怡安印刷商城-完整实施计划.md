## 1. 项目概述

### 1.1 项目背景

怡安印刷商城是河南怡安智能印刷集团旗下的 **B2B全品类商务印刷定制电商平台**，覆盖名片、画册、包装盒、标签、广告物料等 500+ 款产品。核心卖点包括 24 小时极速出货、误活免单承诺、VIP 分级折扣、一键翻单、云仓管理、线下门店 O2O、电商平台代发等。

### 1.2 业务范围（20个模块 / 666个FR / 141个API）

| 优先级 | 模块 | 核心功能 |
|:------:|------|---------|
| P0 | 门户与首页 | Banner、分类、热销榜、新品专区、3D预览 |
| P0 | 用户系统 | 注册/登录/找回密码、API Token（Sanctum）、多设备管理 |
| P0 | 商品系统 | 11大品类、参数模板、动态UI、实时计价、文件上传 |
| P0 | 购物车 | 加入购物车、规格区分、数量修改、结算跳转 |
| P0 | 订单系统 | 普通/批量/云仓/一键翻单、双状态机、多级定金 |
| P0 | 支付系统 | 扫码支付/余额支付/对公转账、回调处理、预存款 |
| P0 | 物流与配送 | 最优物流推荐、运费计算、多地址配送、轨迹同步 |
| P0 | 消息通知 | 站内信/短信/微信模板消息、全流程通知、角标 |
| P0 | 企业认证 | 企业认证（6种状态）、合法性审查、电子签约 |
| P0 | 发票管理 | 普票/专票、7个发票状态、开票审核、电子发票 |
| P0 | 售后服务 | 退货退款/补印/优惠货款、11种原因、三套状态码 |
| P0 | 管理后台 | 客户/VIP/订单/商品/物流/财务/云仓管理 |
| P1 | 会员与VIP | Lv0~Lv8等级、样品/印品折扣、子账号权限 |
| P1 | 收藏与复购 | 产品收藏、常用订单一键翻单 |
| P1 | 客服与帮助 | AI客服、FAQ、印刷知识库 |
| P1 | 样品订单 | 样品专区、独立折扣、简化流程 |
| P1 | 客户投诉与工单 | 5类投诉、自动分配、SLA预警 |
| P1 | 线下门店与O2O | 门店注册、自营物流、客户经理上门 |
| P1 | 电商平台代发 | 淘宝/拼多多/抖音/1688对接、自动发货 |
| P2 | 积分与成长值 | 积分获取/消费/兑换、成长值等级、积分商城 |

### 1.3 架构决策

- **架构风格**: 模块化单体（Modular Monolith），按 `app/Domains/xxx` 划分边界
- **演进路线**:
  - Phase 1（当前）: 单体模块化，Event/Queue 解耦
  - Phase 2（未来）: 某 Domain > 5,000 行时提取为 Laravel Package
  - Phase 3（未来）: 需独立扩缩容时增加 HTTP/gRPC 网关层
- **核心原则**:
  - Controller 必须"薄"，单方法不超过 15 行
  - 业务逻辑写在 Action 类（单一职责）
  - 富模型设计（Eloquent Model 封装状态机、业务规则）
  - 跨 Domain 写操作必须通过 Event + Queue 异步解耦
  - 禁止跨 Domain 直接 `use` 其他 Domain 的 Model

#### 索引设计约束（所有Migration必须遵守）

- 每表必须有自增主键 id
- 业务唯一键必须加 UNIQUE 索引（如 order_no、transaction_no）
- 所有外键必须加索引
- 所有 created_at 建立索引
- 复合索引命名：idx_{table}_a_b
- 唯一索引命名：uk_{table}_{column}
- 金额字段全部使用 DECIMAL(12,2) 或 DECIMAL(15,2)，禁止 FLOAT/DOUBLE
- 敏感字段（手机号、姓名、地址）使用 TEXT 类型（加密后存储）

#### 6.3.2.1 时区与货币处理规范

**时区**：
- 系统统一使用 `Asia/Shanghai`（东八区），数据库、PHP、Nginx、Cron全部配置为同一时区
- 数据库存储：所有时间字段使用 `DATETIME(3)`（含毫秒），不存储时区信息（统一按东八区解释）
- API响应：ISO 8601格式 `2026-05-30T14:30:00+08:00`
- 前端显示：按用户设备时区自动转换（目前仅支持中国时区，未来扩展国际化）
- 定时任务：Crontab按服务器本地时间执行（东八区）

**货币处理**：
- 存储：全部以“分”为单位存储（INT），避免浮点精度问题；`Money`值对象封装分和元的转换
- 显示：前端统一格式 `¥1,234.56`，千分位分隔，保留2位小数
- 计算：PHP端使用 `bcmath` 或 `Money`值对象进行精确计算，禁止直接浮点运算
- 汇率：目前仅支持人民币（CNY），未来扩展多货币时引入汇率服务
- 精度：四舍五入到分（标准舍入），特价场景允许到厘（DECIMAL(15,3)）但展示时四舍五入到分

#### 6.3.4a 分层限流与熔断策略（Rate Limiting and Circuit Breaker）

**分层限流矩阵**：

| 层级 | 工具 | 规则 | 触发后行为 |
|------|------|------|-----------|
| WAF层 | 阿里云WAF / 腾讯云EdgeOne | CC攻击防护：单IP>1000req/min自动封禁10min；异常UA拦截 | 返回403，记录安全日志 |
| CDN层 | 阿里云CDN | 单IP>500req/min限流；非浏览器UA降速 | 返回429，边缘缓存静态页 |
| Nginx层 | nginx-limit-req | 单IP 100req/s突发200；静态资源不限制 | 返回503，记录access_log |
| 应用层 | Laravel Throttle + Redis | 见下表差异化限流 | 返回429 + Retry-After头 + 降级内容 |
| 用户层 | Redis用户ID桶 | VIP企业用户配额翻倍；黑名单用户全局拒绝 | 返回429，提示联系客服 |

**API差异化限流规则**：

| API类型 | 匿名 | 认证用户 | 限流响应 |
|---------|------|---------|---------|
| 搜索/浏览 | 120/min | 600/min | 降级为固定热门结果 |

**熔断器策略（Circuit Breaker）**：

| 服务 | 失败阈值 | 熔断时长 | 半开探测 | 降级策略 |
|------|---------|---------|---------|---------|
| 库存服务 | 5s内失败率>50% | 30s | 每10s放1请求 | 显示"库存查询中，请稍候"，缓存兜底 |

| 物流查询 | 连续3次超时 | 120s | 每30s放1请求 | 返回缓存轨迹，显示"更新中" |
| 短信服务 | 连续5次失败 | 60s | 每20s放1请求 | 降级邮件通道 |

**限流响应规范**：HTTP 429 + `Retry-After: {秒数}` + `X-RateLimit-Reset: {时间戳}`；前端收到429后显示"操作过于频繁，{秒数}秒后重试"，支付类场景引导用户联系客服。

#### 6.4b 分布式事务管理规范（Distributed Transaction Specification）

**本地事务优先**：单Domain内操作使用数据库本地事务（Laravel DB::transaction），禁止长事务（执行时间<3s）。

**跨Domain事务——Outbox Pattern**：
- **Outbox表**：`outbox_messages`（id, aggregate_type, aggregate_id, event_type, payload, status[pending/sent/failed], retry_count, created_at, sent_at）
- **写入流程**：业务操作与Outbox记录在同一本地事务中写入，确保"业务数据提交 = 消息持久化"
- **投递流程**：Cron每30s扫描`status=pending`的Outbox记录，投递到Redis队列后标记为sent；失败3次后标记failed并告警
- **幂等保障**：Consumer端通过`event_id`（UUIDv7）去重，Redis SETNX 24h

**Saga轻量级补偿（跨Domain长事务）**：

| Saga步骤 | 正向Action | 补偿Action | 触发补偿条件 |
|---------|-----------|-----------|------------|
| 1.创建订单 | CreateOrderAction | CancelOrderAction | 库存预占失败/支付超时 |
| 2.预占库存 | ReserveInventoryAction | ReleaseInventoryAction | 支付失败/订单取消 |
| 3.创建支付 | CreatePaymentAction | ClosePaymentAction | 支付网关异常/用户取消 |
| 4.扣减库存 | DeductInventoryAction | ReturnInventoryAction | 发货前订单取消 |
| 5.创建物流 | CreateShipmentAction | CancelShipmentAction | 发货前订单取消 |

**补偿执行规则**：
- 补偿Action必须是幂等的（同一order_no多次执行结果一致）
- 补偿顺序与正向相反（LIFO）：先取消物流→返还库存→关闭支付→取消订单
- 补偿失败时进入`failed_saga_compensations`表，人工介入处理
- 补偿超时：单步补偿>30s视为失败，记录告警并转人工

**最终一致性校验**：
- 每日T+1对账：订单表 vs 库存流水 vs 支付流水 vs 物流表，四方数据一致性校验
- 差异记录到`reconciliation_daily`表，>¥0.01触发P2告警

**Event-Listener Mapping Rules:**
- Synchronous Events: Triggered in same request (e.g., OrderCreated -> ReserveInventoryListener)
- Async Queue Events: Listener implements ShouldQueue (e.g., PaymentSuccess -> SendNotificationListener)
- Broadcast Events: Implement ShouldBroadcast for WebSocket push

#### 缓存设计规范（Cache Design）

| 规范项 | 要求 |
|--------|------|
| Key命名 | `yian:{domain}:{identifier}:{sub}`，如 `yian:product:123:detail` |

### 8.2 计价Pipeline步骤（Pricing Pipeline Steps）

```
PricingInput(category, size, material, process, quantity, vipLevel, couponCode, isUrgent, packaging)
  -> Step 1: BasePriceLookup (query base price from product)
  -> Step 2: MaterialProcessMarkup (accumulate material and process surcharges)
  -> Step 3: QuantityDiscount (apply tiered pricing based on quantity)
  -> Step 4: VipOrSampleDiscount (apply VIP or sample discount, mutually exclusive)
  -> Step 5: CouponDiscount (apply coupon, stackable with VIP)
  -> Step 6: UrgentPackagingFee (add urgent fee and packaging fee)
  -> FloorCapValidation (minimum 0.5x of original price)
  -> PricingResult(Money total, Money breakdown[], Money savings)
```

| 按商品类型 | 印刷品 vs 成品包装物 vs 辅料 | 同上 | 画册+包装盒，包装盒为外购成品 |
| 按客户要求 | 客户主动要求分批发货 | 同上 | "先发画册，名片下周发" |

**拆分后规则**：
- 主订单（Parent）保留总金额/总优惠/支付信息，子订单（Sub-order）继承主订单支付状态
- 运费按子订单独立计算，主订单运费=各子订单运费之和（合并发货时可减免）
- 优惠券按商品金额比例拆分至子订单；积分按客户指定或平均拆分
- 子订单独立状态机（FM/nM），但客户侧仍看到统一订单视图
- 补印（reprint）作为独立子订单（order_type=5），关联原订单

**合并发货条件**（同时满足）：
- 同一客户（customer_id）
- 同一收货地址（address_id）
- 同一工厂/仓库（factory_id/warehouse_id）
- 均未发货（status < shipped）
- 客户主动申请合并或系统默认合并（24h内下单）

**合并后规则**：
- 新物流单号回填至各子订单；原独立运费按最优方案重新计算，差额退还余额
- 合并操作记录审计日志，客户可在发货前取消合并

- **温湿度控制**：夏季高温车厢>40°C时，画册/不干胶需加隔热层；雨季湿度>80%时加干燥剂

**包装费计价规则**：
- 标准包装（纸箱+PE袋+填充）：¥2-5/件（按体积阶梯）

### 14.1.1 RBAC权限模型（RBAC Permission Model）

**角色定义**：
| 角色 | 权限范围 | 数据隔离 |
|------|----------|----------|
| 超级管理员 | 全部功能 | 全部数据 |
| 运营经理 | 商品/订单/客户/营销/内容 | 全部数据 |
| 客服人员 | 客户/订单/售后/工单 | 仅分配客户 |
| 仓库管理员 | 订单发货/库存/物流 | 订单脱敏（隐藏价格） |
| 财务人员 | 支付/退款/发票/对账 | 全部支付数据 |
| 审计人员 | 审计日志/操作记录 | 全部日志，只读 |

**权限粒度**：
- 菜单级：控制左侧导航显示/隐藏
- 按钮级：控制页面内操作按钮（新增/编辑/删除/导出/审核）
- 字段级：控制列表/详情页字段显示（如仓库角色不显示成本价）
- 数据级：SQL查询时附加数据范围条件（如客服仅看customer_service_id=自己的数据）

**实现方式**：
- 角色-权限：多对多，permissions表存储权限标识符（如order:read, order:write, order:delete）
- 用户-角色：多对多，中间表user_roles
- 数据隔离：AOP拦截Query Builder，自动注入数据范围条件
- 前端控制：路由meta.roles + 按钮v-permission指令 + 字段show/hide配置

### 16.4 生产环境部署架构（Production Deployment Architecture）

#### 服务器拓扑

| 角色 | 数量 | 配置 | 说明 |
|------|:----:|------|------|
| LB (Nginx) | 2 | 4C8G | 负载均衡+SSL终端+静态资源缓存；主备模式 |
| App (Octane) | 4 | 8C16G | Laravel Octane Swoole Worker；无状态 |
| MySQL | 3 | 8C32G+SSD | 1主2从异步复制；读写分离 |
| Redis | 6 | 4C8G | Cluster 3主3从；持久化RDB+AOF |
| Meilisearch | 1 | 4C8G | 单节点（初期）；预留集群扩展 |
| MinIO | 4 | 4C16G+HDD | 分布式纠删码EC:2；2副本 |
| Queue Worker | 2 | 4C8G | Horizon队列消费者；独立部署 |
| Monitoring | 1 | 4C8G | Prometheus+Grafana+Alertmanager |

#### 网络架构

```
Internet
  → WAF (云WAF: 阿里云/腾讯云)
    → CDN (静态资源/图片缓存)
      → Nginx LB (443/80)
        → DMZ (公网可访问)
          → App Cluster (Octane Workers)
            → Internal Network
              → MySQL Master/Slave
              → Redis Cluster
              → Meilisearch
              → MinIO
              → Monitoring
```

- **VPC划分**：公网子网（LB/Nginx）、应用子网（App/Worker）、数据子网（DB/Redis/Meilisearch/MinIO）
- **安全组**：数据库仅允许应用子网访问（3306/6379/7700/9000）；管理后台仅允许VPN/堡垒机IP访问
- **管理后台**：`/api/admin/v1` 和 Admin前端限制公司IP白名单或VPN接入
- **会话保持**：API端无状态（Sanctum Token），无需会话保持；Admin Session端使用Redis存储Session，Nginx ip_hash保证同一IP路由到同一Worker（Octane Swoole模式下Worker间不共享内存）

#### 蓝绿部署SOP

| 步骤 | 操作 | 时长 |
|------|------|:----:|
| 1 | 部署新版本到Blue环境（并行于Green生产环境） | 5min |
| 2 | Blue环境健康检查（HTTP 200 + 业务探针） | 2min |
| 3 | 数据库Migration执行（向前兼容，无破坏性变更） | 3min |
| 4 | 流量切换：Nginx权重 100% Green → 1% Blue → 50% → 100% Blue | 5min |
| 5 | 观察10分钟（错误率<1%，P99 RT正常） | 10min |
| 6 | 切换完成，Green环境保留30分钟作为回滚备选 | 30min |

**回滚触发条件**：错误率>1% / P0告警 / 核心功能异常
**回滚执行**：Nginx权重切回Green，Blue环境下线排查
**DB回滚**：仅支持向前兼容的变更（加字段/加索引），不支持删字段/改类型的破坏性变更

#### 回滚详细SOP（Rollback SOP）

| 步骤 | 操作 | 负责人 | 时长 |
|:--:|------|:------:|:----:|
| 1 | 触发判断：错误率>1% 或 P0告警 或 核心功能异常 | 值班DevOps | 1min |
| 2 | 一键回滚：执行 `deploy/rollback.sh {version}`，Nginx权重切回Green | 值班DevOps | 2min |
| 3 | 验证回滚：Green环境健康检查（HTTP 200 + 业务探针通过） | 值班DevOps | 2min |
| 4 | 通知干系人：钉钉群通知项目组，暂停新功能发布 | 值班DevOps | 1min |
| 5 | 保留现场：Blue环境不下线，保留日志和进程状态供排查 | 值班DevOps | - |
| 6 | 问题排查：DevOps+后端联合排查根因 | 开发团队 | <=4h |
| 7 | 修复验证：修复后Staging验证，再次蓝绿部署 | 开发团队 | - |
| 8 | 复盘会议：24h内产出Postmortem，更新知识库 | 架构师 | 24h内 |

**回滚脚本要求**：
- `rollback.sh`必须幂等，多次执行结果一致
- 自动回滚Nginx配置、代码版本、环境变量
- DB不自动回滚（向前兼容保障），如需回滚DB需人工确认
- 保留Blue环境30min后自动清理（除非标记保留排查）

### B.5 Forbidden Patterns

| 禁止模式 | 替代方案 | 原因 |
|-----------|-------------|--------|
| `ServiceImpl` | `Action` or `Service` (no interface if single implementation) | PHP does not need interface separation by default |
| Direct `update(['status' => X])` | `$model->transitionTo(Status::X)` | Bypasses state machine validation |
| Direct `DB::table()` in business layer | Use Eloquent Model or Repository | Maintainability and consistency |
| `float` for money | Use `Money` value object or `DECIMAL` | Floating point precision loss |
| Cross-Domain Model direct use | Use Repository for read, Event for write | Maintain domain boundaries |
| Logic in Controller | Move to Action/Service | Thin controller principle |
| N+1 queries | Use `with()` eager loading | Performance |

    };

// 7. #[\NoDiscard]（PHP 8.5）— 强制处理返回值，防止遗漏库存释放
final class InventoryService
{
    #[\NoDiscard]
    public function reserve(array $items): ReservationResult
    {
        // 必须处理返回值（成功/失败/部分成功）
        return $this->reserver->execute($items);
    }
}

// 8. Closures in Constant Expressions（PHP 8.5）— 路由/策略常量中定义闭包
final class PricingPolicy
{
    public const \Closure MIN_ORDER_CHECK = fn(Money $amount): bool
        => $amount->cents >= 1000_00; // 最低起订金额 ¥1000
}

// 9. array_first() / array_last()（PHP 8.5）— 简化数组边界取值
public function getFirstProducingFactory(array $factories): ?Factory
{
    return array_first(
        $factories,
        fn(Factory $f) => $f->status === FactoryStatus::Producing,
    ); // 找不到时返回 null，无需 isset($factories[0]) 判空
}
